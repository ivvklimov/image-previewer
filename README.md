# Image Previewer

Сервис для создания превью изображений - HTTP-прокси, который загружает изображения с удалённых серверов, масштабирует и обрезает их до заданных размеров, кэширует результат на диске с использованием LRU-алгоритма и отдаёт клиенту.

Разработан в рамках финального проекта курса Otus Golang Developer Professional.

## Возможности

- **HTTP-прокси**: Загрузка изображений с произвольных upstream-серверов.
- **Ресайз**: Режим `fill` (масштабирование + обрезка по центру без искажения пропорций, алгоритм Lanczos).
- **Поддержка форматов**: JPEG и PNG с автоматическим определением по magic bytes (защита от вредоносных файлов).
- **Дисковый кэш**: LRU-алгоритм вытеснения с лимитом в байтах и атомарной записью файлов.
- **Восстановление состояния**: Корректная инициализация LRU-очереди при холодном старте на основе времени модификации файлов.
- **Проксирование заголовков**: Передача заголовков клиента на upstream с фильтрацией hop-by-hop заголовков (Connection, Host, Transfer-Encoding) согласно RFC 7230.
- **Комплексное интеграционное тестирование**: Изолированное Docker-окружение с HTTPS-upstream, покрывающее полный пайплайн (проксирование, ресайз, кэш, обработка ошибок) в 10+ сценариях без риска для основной среды.
- **Graceful shutdown**: Корректное завершение работы с таймаутом 3 секунды через `signal.NotifyContext`.
- **Версионирование**: Встраивание метаданных (Release, BuildDate, GitHash) через `-ldflags`.
- **Структурированное логирование**: Обёртка над `slog` с выводом запросов в формате Apache Common Log.

### Реализовано на 1 этапе разработки:
- **Проектная структура**: Стандартизированная структура Go-проекта (cmd/, internal/), go.mod, Makefile с таргетами build/run/test/lint, .golangci.yml с настроенными правилами. Архитектура полностью синхронизирована с сервисом календаря: бизнес-логика вынесена в internal/server/http/ (handlers.go, middlewares.go, server.go).
- **Конфигурация**: Дженерик-загрузчик Load[T], приоритет ENV > YAML > defaults (12-Factor App), валидация ValidateAPI (fail-fast), поддержка таймаутов и лимитов кэша.
- **Логирование**: Унифицированная обёртка над slog (internal/logger) с уровнями (debug/info/warn/error), serviceName, выводом в io.Writer для тестов. Middleware LoggingMiddleware логирует каждый запрос в формате Apache Common Log: IP [timestamp] METHOD PATH PROTO STATUS SIZE LATENCY "UA" - статус и клиент фиксируются автоматически, бизнес-события (preview requested) логируются отдельно на уровне INFO/WARN.
- **Версионирование**: Встраивание метаданных через -ldflags (Release, BuildDate, GitHash), поддержка --version и аргумента version, вывод версии в логах при старте/остановке.
- **HTTP-сервер**: Корректная обработка маршрута /fill/{width}/{height}/{url...} по ТЗ, авто-добавление https://, валидация параметров (положительные размеры, валидный URL), таймауты (Read/Write/Idle из конфига), graceful shutdown через signal.NotifyContext с таймаутом 3 секунды.
- **Тестирование**: Table-driven юнит-тесты хендлера (internal/server/http/handlers_test.go) с хелперами (newTestRequest), изоляцией логов через bytes.Buffer, запуск с -race.
- **Инфраструктура**: Инициализация директории кэша (os.MkdirAll), готовность к интеграции прокси/ресайза/дискового кэша на следующих этапах.

### Реализовано на 2 этапе разработки:
- **Прокси-клиент**: Реализован безопасный HTTP-клиент (internal/proxy/fetcher.go) с жесткими таймаутами и лимитом размера тела ответа. Проксирование заголовков: исходные заголовки клиента передаются на upstream-сервер с корректной фильтрацией hop-by-hop заголовков (Connection, Host и др.) согласно RFC 7230. Строгая валидация MIME-типа (только image/jpeg и image/png). Корректный маппинг ошибок upstream (404 -> 404 Not Found, 5xx -> 502 Bad Gateway).
- **Ресайзер**: Реализован алгоритм fill (internal/resizer/resize.go) с масштабированием и обрезкой по центру без искажения пропорций (Lanczos). Безопасность: строгая проверка magic bytes до декодирования (защита от вредоносных файлов). Сохранение формата: изображение кодируется обратно в тот же формат, в котором было получено (JPEG -> JPEG, PNG -> PNG).
- **Дисковый кэш**: Реализовано хранилище (internal/cache/disk.go) с генерацией ключей через SHA256 (url:width:height). Атомарная запись (сначала в .tmp, затем os.Rename) для защиты от повреждения файлов при краше. Используется подход Content-Addressable Storage: файлы сохраняются без расширений, формат определяется по бинарному содержимому.
- **HTTP-интеграция**: Полная сборка пайплайна в HandlePreview: проверка кэша (Cache Hit) -> запрос к upstream при промахе -> ресайз -> сохранение результата в кэш -> отдача клиенту с корректным Content-Type (автоматически определяется через http.DetectContentType).
- **Тестирование**: Написаны comprehensive unit-тесты для кэша, прокси (включая проверку проксирования заголовков и фильтрации hop-by-hop) и ресайзера (проверка magic bytes, edge-cases). Реализован интеграционный тест хендлера с поднятием тестового upstream-сервера (httptest.NewTLSServer). Все тесты стабильно проходят с флагом -race.

### Реализовано на 3 этапе разработки:
- **LRU-алгоритм**: Базовая архитектура (двусвязный список в связке с map для обеспечения O(1) на всех операциях) адаптирована из ранее реализованного и проверенного домашнего задания (hw04_lru_cache). Изначально in-memory реализация адаптирована для работы с дисковым хранилищем: добавлены атомарная запись файлов, физическое удаление при вытеснении и восстановление состояния при холодном старте.
- **Лимит в байтах**: В отличие от лимита по количеству элементов, кэш ограничен суммарным размером данных в байтах (настраивается через limit_bytes в config.yaml). При каждом Set пересчитывается currentSize, и при превышении лимита запускается вытеснение старых элементов.
- **Вытеснение наименее недавно использованных**: Метод evict() удаляет элементы из конца LRU-очереди (Back()), пока currentSize ≤ maxSize. Файлы физически удаляются с диска через os.Remove.
- **Восстановление состояния при холодном старте (restoreFromDisk)**: При запуске сервера сканируется директория кэша, файлы сортируются по времени модификации (ModTime) и добавляются в конец LRU-очереди через PushBack. Это гарантирует, что "старые" файлы (не запрашивавшиеся текущим экземпляром сервера) будут вытеснены в первую очередь. Если после восстановления суммарный размер превышает лимит - сразу запускается вытеснение.
- **Потокобезопасность**: Используется sync.RWMutex - Lock для операций записи/модификации (Get, Set, evict), RLock для операций чтения (Has, Stats). Протестировано через стресс-тест с двумя горутинами (по 10 000 операций каждая) с флагом -race.
- **Тестирование**: Написаны comprehensive unit-тесты, покрывающие вытеснение по байтам, обновление LRU-порядка при чтении, восстановление состояния при холодном старте (включая сценарий, когда после рестарта размер уже превышает лимит) и конкурентный доступ. Все тесты стабильно проходят с флагом -race.
- **Инфраструктура и качество кода**: Добавлен конфигурационный файл .golangci.yml с набором линтеров (godot, depguard, errcheck, gofmt, govet, staticcheck, revive, unused). Исправлены все замечания линтера: разрешены необходимые внешние зависимости в depguard, устранены переопределения встроенных функций (max -> maxSize), неиспользуемые параметры в тестах заменены на _. Все Go-файлы сконвертированы из CRLF в LF для единообразия кодировки окончаний строк.
- **Визуальная демонстрация**: Прикреплены скриншоты работы сервера: вывод ls -lahrt --time-style, показывающий вытеснение файлов по LRU при сохранении лимита; логи сервера с демонстрацией полного цикла: инициализация кэша (limit_bytes=102400), обработка cache miss/cache hit, корректная обработка ошибок upstream (context deadline exceeded -> 502 Bad Gateway без падения сервиса, последующие успешные повторные запросы).

### Реализовано на 4 этапе разработки:
- **Интеграционное тестирование**: Развёрнуто полноценное изолированное тестовое окружение на базе Docker Compose. В качестве upstream-сервера используется Nginx, раздающий набор тестовых изображений (JPEG/PNG) из директории fixtures/. Реализовано 10 интеграционных тестов, покрывающих все 7 обязательных сценариев из ТЗ плюс 3 дополнительных.
- **HTTPS для upstream**: Настроен Nginx на HTTPS (порт 443) с self-signed сертификатом. Сгенерирован сертификат с SAN DNS:upstream,DNS:localhost командой openssl req -x509 для совпадения с именем сервиса в Docker-сети.
- **Разделение Dockerfile**: Создан отдельный Dockerfile.integration для тестового окружения, содержащий тестовый CA-сертификат в доверенных (/usr/local/share/ca-certificates/). Production-образ (Dockerfile) остаётся чистым - без тестовых артефактов, что соответствует принципам безопасности и минимизации размера образа.
- **Изоляция проектов через COMPOSE_PROJECT_NAME**: Все команды Makefile для интеграционных тестов используют COMPOSE_PROJECT_NAME=image-previewer-test, а для основного приложения - image-previewer-app. Это гарантирует полную изоляцию: контейнеры, сети и тома имеют разные префиксы, и команда down -v в тестах физически не может затронуть рабочие тома с данными или production-окружение.
- **Makefile**: Структура с чётким разделением команд:
  - **Локальная разработка**: make build, make run-local, make test, make lint.
  - **Docker-приложение**: make docker-build, make docker-run (make run), make docker-rebuild, make docker-stop, make docker-clean, make docker-logs (make logs).
  - **Интеграционные тесты**: make integration-test (полный цикл), make integration-test-run (быстрая итерация), make integration-test-build, make integration-test-up, make integration-test-down.

### Реализовано на 5 этапе разработки:
- **CI/CD Пайплайн**: Настроен многоэтапный workflow (.github/workflows/ci.yml), который запускается при создании Pull Request, push в main/master и создании релизных тегов (v*). Пайплайн гарантирует, что код проверяется до попадания в основную ветку, а релизы создаются только после успешного прохождения всех стадий.

- **Статический анализ (Linting)**: Интегрирован golangci-lint (стабильная версия v1.64.8). Конфигурация .golangci.yml полностью актуализирована: удалены неподдерживаемые линтеры, что обеспечивает совместимость и чистый проход проверок без ошибок.

- **Надёжное Unit-тестирование**: Реализован двухэтапный запуск тестов для максимальной информативности и стабильности:

    - go test -race -count=100 - стресс-тестирование для гарантированного выявления race conditions и нестабильных (flaky) тестов.

- **Автоматизация релизов**: Реализован отдельный job release, который срабатывает при создании тега (например, v1.0.0). Он автоматически скачивает собранный артефакт и создаёт GitHub Release с прикреплённым архивом и сгенерированным changelog, используя принцип минимальных привилегий (permissions: contents: write).
